# Git Changes Manager

A local-only web app that lists the files changed in a Git repository on your machine and lets you triage them on a **Trello-style drag-and-drop board**. No diffs, no file contents — just filenames and your review state.

Four columns:
- **Uncategorized**
- **Needs More Review**
- **Reviewed**
- **Ignore for Now**

Drag a file card between columns to set its category. Assignments persist in a local `categories.json` file (no database).

The **backend** is zero-dependency Node.js (`server.js`). The **frontend** is a React + Vite + dnd-kit board that builds into `public/`, which the Node server serves.

## Requirements
- [Node.js](https://nodejs.org) (any recent version).
- Git on your PATH. Verify with:
  ```
  git --version
  ```

## Run (normal use)
The frontend is already built into `public/`, so you only need the server:
```
node server.js
```
Then open <http://localhost:3000>.

If you change frontend code, rebuild it first:
```
cd frontend
npm install      # first time only
npm run build    # outputs to ../public
cd ..
node server.js
```

To use a different port:
```
# macOS/Linux
PORT=4000 node server.js
# Windows PowerShell
$env:PORT=4000; node server.js
```

## Develop (hot reload)
Run the backend and the Vite dev server in two terminals. The dev server proxies `/api/*` to the backend.
```
# terminal 1 — backend
node server.js

# terminal 2 — frontend dev server
cd frontend
npm install      # first time only
npm run dev
```
Open the URL Vite prints (usually <http://localhost:5173>). Edit files in `frontend/src/` and the page hot-reloads. When done, run `npm run build` to bake changes into `public/` for normal use.

## Use
1. Paste a repository path into the box at the top:
   - Windows: `C:\repos\my-project`
   - macOS/Linux: `/Users/you/my-project`
2. Click **Load** (the path is remembered for next time).
3. Each changed file appears as a card with its status badge (Added / Modified / Deleted / Untracked / Renamed). **Drag cards between columns** to categorize them — each move saves immediately.

The board includes both **staged and unstaged** changes plus **untracked** (newly added) files.

```

  categories.json        # auto-created storage of your category assignments
  public/                # built frontend (generated by `npm run build`)
  frontend/              # React source
    package.json
    vite.config.js       # builds to ../public; dev-proxies /api to :3000
    index.html
    src/
      main.jsx
      App.jsx            # repo input, DndContext, drag handling
      Board.jsx          # the four columns
      Column.jsx         # a droppable column
      Card.jsx           # a draggable file card
      api.js             # calls /api/status and /api/board
      columns.js         # column definitions + category mapping
      styles.css
  README.md
```

## How it works
- The backend runs `git status --porcelain=v1 -z --untracked-files=all` against the path you provide (via `execFile`, so no shell quoting issues with Windows paths or filenames containing spaces).
- Each column maps to a category value: Uncategorized → `""`, Needs More Review → `needs-review`, Reviewed → `reviewed`, Ignore for Now → `ignore`.
- Dropping a card sends `POST /api/board` with the full ordered list, so both the **category and the position** of every card are saved. On reload, files come back in their saved order and columns; the UI reverts if a save fails.
- Assignments are stored per repo in `categories.json` as `{ categories: {path: category}, order: [path, ...] }`, so you can triage multiple repos and they stay separate. (Older files using the flat `{path: category}` shape are still read.)

## Notes
- **Reset all assignments:** delete `categories.json` (it's recreated automatically). Dragging a card back to **Uncategorized** removes just that file's saved category.
- **Cross-platform:** works on Windows, macOS, and Linux. Backslash paths on Windows are fine — the path is passed as a working directory, never through a shell. Porcelain output is locale-independent.
- **Local only:** the server binds to localhost; there is no authentication because it is not meant to be exposed to a network.
